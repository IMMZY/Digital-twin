### Q0.1 ā?" What other files or values should never be committed?

Beyond the Terraform state and provider cache, several other categories of files must be kept out of version control. AWS credentials and `.env` files containing secrets like `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, or any API keys should be listed in `.gitignore` ā?" typically by adding `.env`, `.env.local`, and any `.env.*` variants. The `terraform.tfvars` file is safe in this activity because it holds only non-sensitive configuration, but any future `.tfvars` file containing passwords, tokens, or private keys must be excluded. Lambda deployment artifacts like `lambda.zip` should also not be committed ā?" they are build outputs that can be regenerated and are often large. Python virtual environment folders (`venv/`, `.venv/`, `__pycache__/`) and Node.js `node_modules/` are similarly generated and should be excluded. In general, any file that contains credentials, can be reproduced from source, or is platform-specific has no place in a Git repository. The safest habit is to start every project with a thorough `.gitignore` and to run `git status` carefully before every commit.

---

### Q0.2 ā?" What happens if you delete `terraform.tfstate` and re-apply?

Terraform relies entirely on its state file to know which real AWS resources it is responsible for. If the state file is deleted, Terraform has no memory of what it has already created. Running `terraform apply` again would cause it to attempt to create every resource from scratch, as though none of them existed. For resources with globally unique names ā?" such as S3 buckets or Secrets Manager secrets ā?" this would immediately fail with errors like `BucketAlreadyExists` or `SecretAlreadyExists`. For resources without uniqueness constraints, Terraform might succeed in creating duplicates, resulting in orphaned infrastructure that incurs cost but is no longer managed by Terraform. This is dangerous because it produces an inconsistent environment where some resources are tracked and others are invisible to Terraform, making future `plan` and `apply` operations unreliable. The correct remediation is `terraform import`, which re-associates existing AWS resources with the state file, but this is tedious and error-prone. This is precisely why the state file must be protected ā?" ideally stored in a remote backend such as S3 with DynamoDB locking, never on a local machine or in Git.

### Q1.1 ā?" When would you use a second variable file like `prod.tfvars`?

A second variable file is useful when you want to drive different non-sensitive configuration values into the same Terraform code for different environments without committing those values to a shared file. For example, a `prod.tfvars` might increase `lambda_timeout` to 60 seconds and point `bedrock_model_id` to a higher-capacity model that the production budget can afford. You reference it at apply time with the `-var-file` flag:

```bash
terraform apply -var-file="prod.tfvars"
```

This keeps the configuration DRY ā?" one set of `.tf` files, multiple runtime configurations ā?" while making the differences between environments explicit and reviewable in version control.

---

### Q1.2 ā?" What happens when a variable has no default and no value is provided?

Terraform will pause the `plan` or `apply` and interactively prompt you to type the value in the terminal. If you are running Terraform in a non-interactive context (such as a CI/CD pipeline), the run will fail with an error stating that the variable has no value. This is actually a useful safety feature for required secrets or region settings where a default would be meaningless or dangerous ā?" it forces the operator to make an explicit choice rather than silently falling back to something incorrect.

### Q2.1 ā?" Purpose of `source_code_hash = filebase64sha256("lambda.zip")`

The `source_code_hash` attribute gives Terraform a fingerprint of the deployment ZIP. When you run `terraform apply`, Terraform computes the SHA-256 hash of `lambda.zip` and compares it to the hash stored in the state file from the previous apply. If the hashes differ ā?" meaning the ZIP contents have changed ā?" Terraform knows it must update the Lambda function's code. Without this line, Terraform would look only at the resource's declared attributes (function name, runtime, memory size, etc.) and, since none of those changed, would report "No changes" and skip the code update entirely. Your Python changes would silently never reach the deployed function, which is a frustrating and hard-to-debug situation.

---

### Q2.2 ā?" `aws_iam_role_policy_attachment` vs `aws_iam_role_policy`

`aws_iam_role_policy_attachment` attaches an **AWS managed policy** ā?" a pre-written, versioned policy document that AWS maintains. You reference it by its public ARN. The advantage is that AWS keeps these policies up to date as new service features are released; the disadvantage is that they are often broader than necessary (for example, `AmazonDynamoDBFullAccess` grants every DynamoDB action on every table in the account).

`aws_iam_role_policy` creates a **custom inline policy** that you write yourself in JSON. It is embedded directly in the role rather than existing as a separate IAM resource. This is the right choice when you need fine-grained scope ā?" for example, the Secrets Manager policy in this file restricts `GetSecretValue` to only secrets whose ARN begins with the project prefix, preventing the Lambda from reading secrets belonging to other applications.

In general, use managed policies for standard AWS service permissions where breadth is acceptable, and use inline custom policies when you need to scope permissions tightly to specific resources.

### Q3.1 ā?" Why is `lifecycle { ignore_changes = [secret_string] }` critical?

Without the `ignore_changes` lifecycle block, every `terraform apply` would overwrite the `secret_string` field with the placeholder value `REPLACE_WITH_CLOUDFRONT_URL_AFTER_APPLY`. This is exactly the opposite of what you want: after the first apply, you manually update the secret with the real CloudFront URL so that Lambda's CORS configuration points to the correct domain. If Terraform were allowed to manage `secret_string` freely, the next `terraform apply` ā?" whether triggered by a code change, a team member running a routine update, or an automated pipeline ā?" would silently reset the secret back to the placeholder. Lambda would then read the wrong CORS origin at startup, and the frontend would stop being able to communicate with the backend. The `ignore_changes` directive tells Terraform to create the initial value once and then leave that field alone forever, trusting that out-of-band updates (via the CLI or console) are intentional and authoritative.

---

### Q3.2 ā?" DynamoDB with only a hash key and no sort key

With `session_id` as the sole hash key, each DynamoDB item is uniquely identified by its session ID. This is a direct key-value lookup model: you can retrieve or overwrite any session's conversation history in O(1) time by providing its `session_id`. Because there is no sort key, there is only one item per session ā?" each `put_item` call for the same `session_id` replaces the previous record entirely, which is exactly the right behaviour for a conversation history store where the entire message list is always read and written as a unit.

The trade-off is that you cannot efficiently query across sessions. For example, sorting sessions by `updated_at` or filtering by a specific user would require a full table scan, which is slow and expensive at scale. To support those access patterns you would need a Global Secondary Index (GSI) with `updated_at` as a sort key, or a different schema. For this application ā?" where each request retrieves exactly one session by its known ID ā?" the simple hash-only design is appropriate and keeps both latency and cost at their lowest.

### Q4.1 ā?" Why `origin_protocol_policy = "http-only"` does not create a security problem

There are two separate connections involved in a CloudFront request. The first is between the user's browser and CloudFront, which is always HTTPS ā?" enforced by `viewer_protocol_policy = "redirect-to-https"`. The second is between CloudFront and the S3 origin, which uses HTTP. This inner connection is not a security problem because S3 static website endpoints are owned by AWS and reside entirely within AWS's private network backbone. There is no public internet path between CloudFront's edge nodes and the S3 regional endpoint, so there is no realistic opportunity for a man-in-the-middle attack on that leg of the journey. The user's data is fully encrypted from their browser all the way to CloudFront, which is where the sensitive exposure would occur. The CloudFront-to-S3 segment is an internal AWS infrastructure hop. This is a well-established and accepted CloudFront architecture pattern.

---

### Q4.2 ā?" Why `custom_error_response` redirecting 403/404 to `/index.html` is required for SPAs

A Next.js application built with `output: 'export'` is a Single-Page Application where client-side JavaScript handles all routing. The actual HTML file that exists in the S3 bucket is only `index.html` at the root ā?" there is no physical `/chat/index.html` file. When a user navigates directly to `https://your-twin.cloudfront.net/chat` (for example, by typing it in the browser or refreshing the page), CloudFront asks S3 for `/chat`, finds no such object, and S3 returns a 403 (the bucket's policy returns 403 for missing keys) or 404. Without the `custom_error_response` blocks, the user would see a raw AWS XML error page instead of the application. By intercepting these errors and serving `index.html` with a 200 status code, CloudFront delivers the SPA shell, which then reads the URL, runs the JavaScript router, and renders the `/chat` view correctly. This is the standard pattern for deploying any SPA ā?" React, Next.js, Vue ā?" on S3 and CloudFront.

### Completed `secrets.py`

```python
import boto3
import json
import os

_cache = {}


def get_secret(secret_name: str) -> dict:
    """
    Retrieve a secret from AWS Secrets Manager and return it as a Python dict.
    Results are cached in memory for the lifetime of the Lambda execution environment.
    """
    if secret_name in _cache:
        return _cache[secret_name]

    region = os.getenv("AWS_REGION", "us-east-1")

    # Blank A ā?" create boto3 secretsmanager client
    client = boto3.client("secretsmanager", region_name=region)

    try:
        # Blank B ā?" call get_secret_value
        response = client.get_secret_value(SecretId=secret_name)

        secret_string = response["SecretString"]
        secret_dict = json.loads(secret_string)

        _cache[secret_name] = secret_dict
        return secret_dict

    except Exception as e:
        print(f"Warning: could not retrieve secret '{secret_name}': {e}")
        return {}
```

### Q5.3 ā?" Lambda Cold Starts and Lazy Initialisation

A Lambda "cold start" occurs when AWS needs to create a fresh execution environment for a function ā?" either because no warm environment exists yet (first invocation after deployment or a period of inactivity) or because traffic spikes require additional capacity. During a cold start, AWS provisions a container, downloads the deployment ZIP, initialises the Python runtime, and runs all module-level code before the handler is called. This initialisation phase adds latency, typically 1ā?"3 seconds for a Python function with boto3.

The lazy initialisation pattern in `_get_table()` exploits the fact that Lambda reuses execution environments across multiple invocations. By storing `_dynamodb` and `_table` as module-level variables and initialising them only once (when `_table is None`), the DynamoDB resource object is created during the cold start and then reused for every subsequent warm invocation without any additional setup cost. Creating a new `boto3.resource("dynamodb")` and `Table` object inside `load_conversation()` on every call would add roughly 10ā?"50 milliseconds of object-creation overhead per invocation, accumulate TCP connection setup costs, and prevent boto3 from reusing its internal connection pool. At scale, these costs add up meaningfully ā?" both in latency and in cost, since Lambda bills for duration in 1ms increments.

### Q8.1 ā?" Why Manual Changes to Terraform-Managed Infrastructure Are Dangerous

When two engineers are working on the same Lambda function ā?" one through Terraform and one through the AWS console ā?" they are operating on the same resource with no shared source of truth. The Terraform engineer's state file reflects what Terraform last applied, not what the console engineer changed. When the Terraform engineer next runs `terraform apply`, Terraform will overwrite the console engineer's changes without warning, because it sees only that the real infrastructure diverges from its own configuration and dutifully corrects it. The console engineer's work is silently lost. In the opposite scenario, if the console engineer makes a change that Terraform doesn't know about and then the Terraform engineer runs a `plan` without reviewing it carefully, they might apply other changes that inadvertently interact badly with the undocumented console change. Over time, the state file drifts further from reality, and `terraform plan` output becomes increasingly difficult to interpret. The fundamental principle is that Terraform-managed resources must be modified exclusively through Terraform ā?" the `.tf` files are the single source of truth, and any out-of-band change is invisible to the rest of the team.

### Q8.2 ā?" Team-Safe Centralised State Storage

Terraform supports **remote backends** for storing `terraform.tfstate` outside the local filesystem. For AWS, the standard approach is the **S3 backend**, which stores the state file in an S3 bucket that all team members have access to. DynamoDB is added alongside S3 to provide **state locking**: when one engineer runs `terraform apply`, Terraform writes a lock record to a DynamoDB table, preventing any other `apply` from running concurrently on the same workspace. This prevents two engineers from simultaneously modifying infrastructure and corrupting the state file with conflicting writes.

The backend configuration in `main.tf` looks like:

```hcl
terraform {
  backend "s3" {
    bucket         = "my-team-terraform-state"
    key            = "twin/dev/terraform.tfstate"
    region         = "us-east-1"
    dynamodb_table = "terraform-state-locks"
    encrypt        = true
  }
}
```

This setup is the baseline for any production team using Terraform: state is centralised, versioned by S3, encrypted at rest, and protected from concurrent writes by DynamoDB locking.

## Part 9: Production Environment

### Q9.1 ā?" Why `terraform plan` in `prod` Shows a Full New Resource Set

Each Terraform workspace maintains its own completely independent state file. The `dev` workspace's state file records all the resources created in Part 6 ā?" the Lambda, API Gateway, DynamoDB table, S3 bucket, and so on. When you switch to the `prod` workspace, Terraform loads the `prod` state file, which is empty because nothing has ever been applied in that workspace. From Terraform's perspective, none of the infrastructure exists yet. It reads the `.tf` files (which are the same files, unchanged), compares them to the empty state, and determines that all resources need to be created.

Resource names differ because `local.name_prefix` evaluates to `"twin-${terraform.workspace}"`. In the `dev` workspace this produces `twin-dev`; in the `prod` workspace it produces `twin-prod`. The DynamoDB table in the `dev` workspace is named `twin-dev-conversations`. In the `prod` workspace it would be named `twin-prod-conversations` ā?" a completely separate table with no shared data.

### Q9.2 ā?" Realistic Scenario Where Isolated Environments Prevent a Serious Incident

Consider a team shipping a new version of the Digital Twin that changes the DynamoDB item schema ā?" for example, adding a `user_id` field to the conversation record. A developer tests the change in the `dev` environment, finds it working, and deploys to Lambda in the `prod` workspace. However, the new Lambda code expects `user_id` to be present on every item it reads from DynamoDB, but the existing production DynamoDB table contains thousands of historical items written by the old code without that field. On the next chat request, the production Lambda reads an old item, finds no `user_id`, and crashes with a `KeyError` ā?" taking the live service down for all users.

With proper environment isolation, this scenario is caught in `dev` before it reaches production. The developer replays real-seeming conversations in `dev`, observes the crash, adds defensive code to handle missing `user_id` fields gracefully, retests in `dev`, and only then promotes to `prod`. The production DynamoDB table and its data were never touched during this debugging cycle. Without isolation ā?" if `dev` and `prod` shared the same infrastructure ā?" the initial test would have corrupted real users' conversation history and caused a live outage.

#### Q-A.1 ā?" Why Output the ARN, Not the Value?

Terraform outputs are stored in plaintext in the state file and printed to the terminal after every apply. If the actual `CORS_ORIGINS` value ā?" or any future secret like a database password or API key ā?" were included as an output, it would be visible in terminal history, CI/CD logs, and the state file, defeating the entire purpose of Secrets Manager. The ARN is a safe identifier that tells you where the secret lives without revealing what it contains. Any system that needs the actual secret value should retrieve it at runtime using the Secrets Manager SDK with appropriate IAM permissions, so that access is logged in CloudTrail and access can be revoked without changing application code.

  }

  alarm_description  = "Alert when the Digital Twin Lambda produces any errors"
  treat_missing_data = "notBreaching"
  alarm_actions      = [aws_sns_topic.alerts.arn]

#### Q-B.1 ā?" Is `threshold = 0` the right setting for production?

For a student project or a low-traffic internal tool, triggering on any single error makes sense ā?" you want to know immediately if anything goes wrong. However, in a production service handling thousands of requests per hour, a threshold of zero can produce alert fatigue. Transient errors are inevitable: a single network timeout, a brief Bedrock throttle, or an occasional malformed client request should not page an on-call engineer at 3 AM. A threshold of 5 errors per 5-minute period is more appropriate when the service has established a baseline and the team wants to be notified only when there is a pattern suggesting a real problem ā?" such as a bad deployment, a service outage, or unexpected traffic ā?" rather than a single isolated hiccup. The right threshold depends on the application's error budget and the service level objective (SLO) agreed with users.

#### Q-C.1 ā?" When Is PITR Clearly Justified?

Imagine a healthcare company using a Digital Twin to capture clinician consultation notes stored in DynamoDB. A developer accidentally deploys a migration script that overwrites the `messages` field on every item in the table with an empty list ā?" wiping months of consultation records. Without PITR, those records are gone permanently and the company faces regulatory violations for losing patient data. With PITR enabled, the team can restore the entire table to its exact state from 30 seconds before the script ran, recovering every record with no data loss. At $0.20 per GB per month, protecting a 10 GB table costs $2.00 per month ā?" a trivial expense compared to the legal, reputational, and operational cost of permanently losing sensitive records. In any domain where data has regulatory, legal, or business-critical significance ā?" healthcare, finance, legal, e-commerce order history ā?" PITR is clearly justified.
